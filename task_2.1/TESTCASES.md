Тест-кейсы для API микросервиса объявлений

Host: `https://qa-internship.avito.com`

API-контракт (из Postman-коллекции)

POST `/api/1/item` - Сохранить объявление
- Request: `{ "sellerID": <integer>, "name": <string>, "price": <integer>, "statistics": { "likes": <integer>, "viewCount": <integer>, "contacts": <integer> } }`
- Response 200: `{ "id": <string>, "sellerId": <integer>, "name": <string>, "price": <integer>, "statistics": { "likes", "viewCount", "contacts" }, "createdAt": <string> }`
- Response 400: `{ "result": { "messages": {...}, "message": <string> }, "status": <string> }`

GET `/api/1/item/:id` - Получить объявление по идентификатору
- Response 200: массив `[{ "id", "sellerId", "name", "price", "statistics": {...}, "createdAt" }]`
- Response 400/404/500

GET `/api/1/:sellerID/item` - Получить все объявления пользователя
- Response 200: массив `[{ "id", "sellerId", "name", "price", "statistics": {...}, "createdAt" }]`
- Response 400/500

GET `/api/1/statistic/:id` - Получить статистику по объявлению
- Response 200: массив `[{ "likes", "viewCount", "contacts" }]`
- Response 400/404/500

> Примечание: В Postman-коллекции поле в запросе называется `sellerID` (capital ID), а в ответе - `sellerId` (camelCase). Это расхождение зафиксировано как возможный баг.

Техники тест-дизайна

- Классы эквивалентности: валидные/невалидные типы и значения для `sellerId` (int/string/negative), `price` (int/string/negative/zero/float), `name` (string/empty/long/special chars)
- Граничные значения: `price = 0`, `sellerId` на границах диапазона (111111, 999999)
- Попарное тестирование: комбинации отсутствующих обязательных полей
- Анализ состояний: create → get → verify (E2E цепочки)
- Таблица решений: отсутствие каждого обязательного поля → ожидаемый код ошибки
- Проверка идемпотентности: POST не идемпотентен (создаёт новый ресурс), GET идемпотентен
